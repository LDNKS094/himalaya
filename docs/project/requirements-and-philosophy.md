# 需求与设计理念

> 本文档记录项目的需求定义、约束条件和贯穿技术选型的指导原则。
> 最终决策结果见 `technical-decisions.md`，决策推理过程见 `decision-process.md`。

---

## 1. 项目定位

Himalaya 是一个基于 Vulkan 的实时渲染器，以光栅化渲染为起步，组织上为日后混合管线的光追和纯路径追踪做考虑。

- **性质**：个人长期学习和练习项目，无明确开发周期限制
- **短期目标**：一年左右完成一个比较写实的画面用于演示
- **场景类型**：通用场景，室内室外都要兼顾
- **开发方式**：AI 辅助开发（极大提升开发效率），瓶颈在于审查、理解和架构决策

---

## 2. 硬件目标与性能约束

### 主要目标平台

- **中端桌面游戏渲染**（非高端、非低端）
- **高端设备的 PCVR**（非独立 VR 头显）

### 性能理念

追求当前硬件技术和渲染技术的**甜蜜点**——性能付出与画面质量的最佳性价比。不必为低端设备过度优化，也不追求高端设备才能运行的技术。

### 平台兼容性

"不关上移动端的门"——对于影响深远的、以后无法通过模块化替换的架构级决策：

- 桌面端不需要高端也能跑得好
- 移动端中高端能跑得了

这不是说要针对移动端优化，而是**架构层面不做出让移动端有过度劣势的选择**。对于可模块替换的具体技术实现不受此约束。

## 3. 技术选型原则

### 原则一：排除过于复杂而收益不高的技术

复杂度和收益必须成正比。如果一个技术的实现成本远高于它带来的视觉或性能收益，排除。

**实际应用**：排除 GPU-Driven Rendering（复杂度弥散到整个架构，场景规模不大时不需要）、排除 Virtual Shadow Maps（实现极复杂，CSM 足以满足需求）。

### 原则二：渐进式实现

先能用，再好用，再优秀。每个模块可按 Pass 1/2/3 分阶段实现：

- **Pass 1**：最简可用版本（如暴力 Forward、基础 Shadow Map）
- **Pass 2**：复杂度相对小、收益大的升级（如 Tiled Forward、CSM）
- **Pass 3**：复杂度在容忍范围内、收益依旧大的进一步提升（如 Clustered Forward）

不是每个模块都必须有三步——有些 Pass 2 才开始做，有些可以跳过中间步骤。Pass 之间的演进应该尽量自然（在已有基础上加东西而非推翻重来）。

### 原则三：业界已验证的技术

采用业界已经被采用过的、有成熟实现和资料的技术。不做实验性方案。资料丰富度直接影响 AI 辅助开发的可靠性。

### 原则四：性能性价比

技术选型始终考虑性能付出与画面质量的比值。同等画面质量下选性能更优的方案；同等性能下选画面更好的方案。

### 原则五：混合管线兼容是加分项

在选择 A 和选择 B 之间，如果 A 能保留到混合管线而 B 不能，可以接受 A 有略微更长的工期或略微更差的性能。但**不为兼容性牺牲感知强烈的当前画质**。

**实际应用**：Forward+ 的 GBuffer 缺失可通过按需生成 thin GBuffer 解决（成本仅在 RT 启用时存在）；烘焙管线的 RT 烘焙阶段与 RT 管线第一个应用场景合并。

### 原则六：不可有明显 glitch

画面可以不精确但不能有很明显的视觉瑕疵。在锐利+闪烁和模糊+不闪烁之间，选择后者。在多画和漏画之间，选择多画。

**实际应用**：CSM texel snapping（消除移动闪烁）、PCSS 配合 temporal filtering（消除采样噪声）、Hardware Occlusion Query 保守两趟策略（零漏画）。

### 原则七：设计约束换技术简洁度

可以接受场景动态灵活性的下降来换取视觉效果和性能的优先。

**实际应用**：限制同时只有一扇可见门可交互（避免 Lightmap 组合爆炸）、多套 Lightmap blend 替代实时 GI（用离线烘焙质量解决通常需要实时 GI 的问题）。

### 原则八：AI 辅助开发下的效率考量

AI 有能力直接写出整个渲染器，瓶颈在于审查理解以及缺乏经验导致的返工。因此：

- 方案的资料丰富度和成熟度很重要
- 架构决策的正确性优于实现速度
- 模块化设计减少返工的影响范围

### 原则九：插件化与延后实现

不是所有东西都需要立刻做。后处理等效果天然是独立的全屏 pass，可插件化设计、独立启用/禁用，排在渲染系统主体完成之后实现。

## 4. 画面质量目标

### 追求方向

在上述所有约束的前提下追求**画面的写实度**。主体是标准 PBR 非透明场景的写实渲染。

### 材质需求

- **主体**：标准 PBR Metallic-Roughness
- **保留能力**：卡通渲染（类似中国二次元游戏风格）的人物渲染（不急于实现）
- **低优先级**：水体、透明半透明、体积渲染——需要有但不是最高优先级

### 视觉偏好

- 不追求绝对精确，但不接受明显的视觉错误
- 宁可模糊也不要闪烁
- 宁可多画（多渲染几个实际被遮挡的物体）也不要漏画（物体突然消失/出现）
- 接受设计约束（如限制可交互元素数量）来换取更简洁的技术方案

## 5. 场景与内容假设

以下假设直接影响了技术选型：

| 假设 | 影响的决策 |
|------|-----------|
| 动态光源不多，会使用烘焙技术 | Deferred 多光源优势被弱化 → Forward+ |
| 动态物体少（静态环境为主） | Lightmap blend 可覆盖大部分 GI 需求 |
| 标准 PBR 为主体着色 | Deferred GBuffer 编码压力小（但仍选了 Forward+） |
| 不做超大开放世界 | 不需要 GPU-Driven、HLOD、World Streaming |
| 昼夜循环需求 | 多套 Lightmap blend、动态天空（Bruneton） |
| 室内室外都有 | GI 方案需兼顾（Lightmap 处理室内，高度雾/大气处理室外） |
| VR 支持（PCVR） | MSAA 重要性提升、带宽敏感性降低（非独立头显） |

## 6. 关键设计理念

### 用离线质量解决实时问题

多套 Lightmap blend 是整个技术栈中最能体现设计理念的决策——用纯离线烘焙的高质量间接光照数据，配合运行时几乎零成本的插值，解决了通常需要复杂实时 GI 系统才能处理的昼夜变化和场景状态切换问题。

### 屏幕空间效果作为性价比之选

SSAO、SSGI、SSR、Contact Shadows 都是屏幕空间效果。它们共同的特点是：实现相对简单、与场景复杂度无关（只和分辨率相关）、共享大量基础设施（深度 buffer、法线 buffer、temporal filtering、Ray March 代码）。缺点是屏幕外信息缺失，但在有 Lightmap 和 Reflection Probes 作为基底的情况下，缺失部分的影响被大幅缓解。

### 基础设施复用

多个看似独立的系统共享底层基础设施，一次建设多处受益：

- **Temporal filtering**：SSAO、SSGI、SSR、AO、SDK upscaler
- **Motion Vectors**：所有 temporal 效果 + DLSS/FSR + Motion Blur
- **屏幕空间 Ray March**：SSR 和 SSGI 共享核心代码
- **BRDF Integration LUT**：IBL Split-Sum 和 Multiscatter GGX 共享生成基础设施
- **深度 PrePass**：Forward+ 标配、所有屏幕空间效果的输入、Hardware OQ 复用
- **SDF 基础设施（条件性）**：如果 GI 引入则阴影、AO 均可受益

### 替换关系而非累加

部分技术之间是明确的替换关系，后一阶段实现后前一阶段退役：

- 高度雾 → Bruneton aerial perspective
- Screen-Space God Rays → Froxel Volumetric Fog
- Hosek-Wilkie → Bruneton（因此直接跳过 Hosek-Wilkie）
- 2D 云层 → 体积云（远期）
- 离线工具烘焙 → RT 烘焙

### 条件性演进路径

部分技术的演进取决于其他系统的存在：

- SDF 阴影混合：仅当 GI 引入 SDF 基础设施时
- Irradiance Probes：仅当 SDF 基础设施已存在时
- WBOIT：仅当简单排序出现实际排序问题时
- Shadow Atlas：仅当光源数量增长到需要调度时
